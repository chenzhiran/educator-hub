7.4 哈夫曼编码 (Huffman Coding)

🌳 贪心魔法与二叉树的完美结晶：

在计算机世界里，文件都是由 0 和 1 组成的。如果我们想把一份巨大的文件“压缩”得更小，该怎么办呢？哈夫曼（Huffman）在1952年提出了一种天才般的贪心策略：让出现频率最高的字符，使用最短的代码；让出现频率极低的字符，使用较长的代码。 这样一平均，总长度就变得特别短！

一、前缀码与最优编码树 (7.4.1)

什么是前缀码？ 如果我们把 A 编码为 0，把 B 编码为 01，当我们收到一串信息 01 时，计算机就懵了：这是 A 和别的东西？还是一个完整的 B？

为了避免这种歧义，我们必须规定：任何一个字符的编码，都不能是另一个字符编码的前缀！ 这种编码被称为“前缀码”。

在图形学中，前缀码完美对应了一棵二叉树：所有的字符都必须老老实实呆在树的叶子节点上。向左走代表 0，向右走代表 1。从树根走到叶子的路径，就是它的编码！

二、哈夫曼算法的构建过程 (7.4.2)

我们要构建一棵最优编码树，哈夫曼给出的贪心策略极其简单粗暴：

策略：每次都从森林中，挑出两个频率最低（最不常用）的节点，把它们合并成一棵新的小树。新树的频率等于它俩的和。如此反复，直到所有的树合并成唯一的一棵大树为止！

🛠️ 代码实现秘籍：常规数组 vs 优先队列

在每次合并时，我们需要找到“当前最小的两个节点”。
如果不讲究技巧，我们可以把节点放在普通的数组里，每次都重新排一次序（或者遍历寻找）。但如果节点非常多，这就太慢了！
此时，优先队列（小顶堆）就成了神兵利器，它会自动帮我们把数字最小的节点浮到最上面！下面为你展示两种写法的对比：

【版本一】基础版：常规数组（每次合并后重新排序）
C++ 基础版 (较慢)
#include <iostream> #include <vector> #include <algorithm> using namespace std; struct Node { int freq; Node *left, *right; }; bool cmp(Node* a, Node* b) { return a->freq < b->freq; } Node* buildHuffmanBasic(vector<int>& freqs) { vector<Node*> nodes; for (int f : freqs) nodes.push_back(new Node{f, nullptr, nullptr}); while (nodes.size() > 1) { // 每次合并前都要重新排序！(极其耗时) sort(nodes.begin(), nodes.end(), cmp); // 取出最小的两个 Node* left = nodes[0]; Node* right = nodes[1]; // 在数组中删除它们 nodes.erase(nodes.begin(), nodes.begin() + 2); // 生成新节点并放回数组 Node* parent = new Node{left->freq + right->freq, left, right}; nodes.push_back(parent); } return nodes[0]; }
Python 基础版 (较慢)
class Node: def __init__(self, freq): self.freq = freq self.left = None self.right = None def build_huffman_basic(frequencies): nodes = [Node(f) for f in frequencies] while len(nodes) > 1: # 每次合并前都要重新排序！(极其耗时) nodes.sort(key=lambda x: x.freq) # 取出最小的两个 left_node = nodes.pop(0) right_node = nodes.pop(0) # 生成新节点并放回数组 parent = Node(left_node.freq + right_node.freq) parent.left = left_node parent.right = right_node nodes.append(parent) return nodes[0]
【版本二】优先队列版：利用最小堆特性（最优解）
C++ 优先队列版 (std::priority_queue)
#include <queue> using namespace std; struct Compare { bool operator()(Node* a, Node* b) { return a->freq > b->freq; // 大于号代表小顶堆 } }; Node* buildHuffmanOptimal(vector<int>& freqs) { // 声明优先队列 (底层自动维护堆结构) priority_queue<Node*, vector<Node*>, Compare> pq; for (int f : freqs) pq.push(new Node{f, nullptr, nullptr}); while (pq.size() > 1) { // 堆会自动弹出最小值，无需我们自己排序！ Node* left = pq.top(); pq.pop(); Node* right = pq.top(); pq.pop(); Node* parent = new Node{left->freq + right->freq, left, right}; // 放回堆中，它会自动“沉淀”到正确的位置 pq.push(parent); } return pq.top(); }
Python 优先队列版 (heapq)
import heapq class Node: def __init__(self, freq): self.freq = freq self.left = None self.right = None def __lt__(self, other): return self.freq < other.freq # 指导heapq比较大小 def build_huffman_optimal(frequencies): pq = [] # 作为堆使用的列表 for f in frequencies: heapq.heappush(pq, Node(f)) # 压入优先队列 while len(pq) > 1: # 堆会自动弹出最小值，无需我们自己排序！ left_node = heapq.heappop(pq) right_node = heapq.heappop(pq) parent = Node(left_node.freq + right_node.freq) parent.left = left_node parent.right = right_node # 放回堆中，它会自动“沉淀”到正确的位置 heapq.heappush(pq, parent) return pq[0]

三、正确性证明的魔法 (7.4.3 & 7.4.4)

为什么把最小的两个合并就一定能得到全局最优秀的编码树呢？这里我们运用上一节学过的 交换论证法 来看看：

1. 贪心选择性质证明：不服就换！

假设存在一棵神秘的最优树 O，在这棵树里，频率最低的两个节点 X 和 Y 竟然没有呆在树的最底层（被别人占据了）。

那我们就强行把占据底层位置的节点，和 X、Y 进行位置互换。因为 X 和 Y 出现的频率更低，把它们放得更深（编码更长），而把高频率的节点提上来（编码变短），这显然只会让总长度变得更小，或者至少保持不变！因此，哈夫曼的选择一定是正确的！

图解 4：交换论证（将低频率节点沉底，总代价必然减少）

2. 最优子结构性质证明：大树包小树

当我们把频率最低的两个节点 X 和 Y 合并成一个新节点 Z 时，原来的 N 个节点问题，就变成了一个 N-1 个节点的子问题。

因为大树的总代价，严格等于子树的总代价 加上 (X的频率 + Y的频率)。这是一个固定的加法！所以，只要子树是最优的，拼出来的大树必然也就是最优的。这完美符合了“最优子结构”的要求！

四、时间复杂度与现实应用 (7.4.5)

哈夫曼编码不仅仅是算法题上的魔法，它在我们的日常生活中无处不在！

比较版本 时间复杂度说明 🌟
基础版本 (数组重排) 一共 N-1 次循环，每次循环都要对数组进行排序。最坏情况下时间复杂度高达 O(N² log N)，非常缓慢。
优先队列版本 一共合并 N-1 次。每次从优先队列中取出和放入数据的代价仅仅是 O(log N)。所以总时间复杂度降至 O(N log N)。效率惊人！空间复杂度为 O(N)。
现实应用：文件压缩 无论是著名的 ZIP (GZIP) 压缩，还是图片压缩格式 JPEG，或者是音频压缩格式 MP3，它们的底层核心步骤中，都深深依赖着优先队列实现的哈夫曼编码，来剥去数据最后的冗余脂肪！
