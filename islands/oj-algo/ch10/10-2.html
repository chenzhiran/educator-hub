10.2 剪枝策略 (Pruning)

✂️ 疯狂生长的杂草树：

我们在 10.1 节见识到了 DFS 的暴力穷举威力，但它的缺点也暴露无遗：时间复杂度是指数级的。如果搜索树长得太大，分支成千上万，计算机就算到冒烟也算不完！

这时候，我们需要高超的剪枝（Pruning）魔法：在向下搜索的过程中，一旦发现某个分支绝对不可能通往出口，或者绝对不可能比已经找到的方案更优，就立刻挥舞剪刀把这根树枝咔嚓剪掉，绝不向下多走一步！

一、剪枝的意义与可行性剪枝 (10.2.1 & 10.2.2)

10.2.1 剪枝的重大意义： 剪枝不是优化某一段小代码，而是直接砍掉整棵子树。如果能提前三层发现错误，就能少运算成百上千次，实现从几天到几毫秒的效率质飞跃！

10.2.2 可行性剪枝： 类似于“走迷宫时看到前方彻底塌方”。在搜索过程中，如果当前状态已经违反了题目的硬性限制条件（比如背包超重了、走出了边界、钱不够了），接下来的所有子分支肯定都完不成。此时，立刻 return 返回！

图解 1：可行性剪枝（超重拦截）的执行过程
C++ 可行性剪枝示范
void dfs(int idx, int currentWeight) { // 🌟【核心：可行性剪枝】 // 一旦发现超重，不再向下递归，直接掉头返回！ if (currentWeight > 10) { return; } if (idx == items.size()) { // 处理合法解的终点 return; } // 尝试装入下一个物品 dfs(idx + 1, currentWeight + items[idx]); // 尝试不装下一个物品 dfs(idx + 1, currentWeight); }
Python 可行性剪枝示范
def dfs(idx, current_weight): # 🌟【核心：可行性剪枝】 # 一旦发现超重，不再向下递归，直接掉头返回！ if current_weight > 10: return if idx == len(items): # 处理合法解的终点 return # 尝试装入下一个物品 dfs(idx + 1, current_weight + items[idx]) # 尝试不装下一个物品 dfs(idx + 1, current_weight)

二、最优性剪枝 (10.2.3)

最优性剪枝的定义： 类似于“买东西凑单时，发现当前买到一半的花费就已经超越了你见过的最低价”。

在求解“求最佳方案、求最少步数、最大价值”等最优解题目时，我们通常会用一个全局变量 bestValue 记录目前已经成功找到的“最好结果”。如果在某条路线的半路上，当前花费的步数就已经大于或等于 bestValue，那么这条路哪怕最后走通了也绝不可能成为新冠军。这时候，立刻剪枝返回！

图解 3：最优性剪枝（步数超越守门员）
C++ 最优性剪枝示范
int bestSteps = 999999; // 全局最强守门员 void dfs(int currentPos, int currentSteps) { // 🌟【核心：最优性剪枝】 // 还没到终点呢，步数就已经追平或超越了已知最优解 // 再往下走也是浪费时间，直接剪枝！ if (currentSteps >= bestSteps) { return; } if (currentPos == targetPos) { bestSteps = min(bestSteps, currentSteps); // 更新纪录 return; } // 继续尝试向左或向右探索分支... }

三、对称性与重复性剪枝 (10.2.4)

重复性剪枝（去重思想）： 绝不在同一层状态中选相同的方案。如果当前要挑出两门课，第一门挑了“数学”，第二门挑了“英语”；回溯回来，另一条分支第一门选了“英语”，第二门又选了“数学”。两个方案在结果上是一模一样的！

为了防止这种换汤不换药的无谓计算，我们往往采用**“规定选取顺序”**或者**“用状态数组进行查重”**的方法，把重复的镜像子树在发芽阶段就直接扼杀！

四、剪枝设计的时机与一般原则 (10.2.5)

在施展剪枝魔法时，教研团队为你梳理了三项基本心法：

1. 越早越好原则 (越靠近树根剪枝效果越惊人)

在 DFS 函数的最开头（拦截器位置）进行剪枝检查，效率是最高的。在第一层砍一刀，能省掉下面整个帝国的计算；如果到了最底层树叶才剪枝，那就只是单纯的判断，毫无节省时间的效果了。

2. 兼顾天平原则 (剪枝判断本身不能太耗时)

剪枝是为了省时间。如果在判定“该不该剪枝”时，你写了一个复杂的循环去验证，结果判定本身花的时间比直接搜还要长，那就成了本末倒置的“负优化”！好的剪枝判定，复杂度应该是常数级的。

3. 正确性原则 (绝不能错杀无辜)

这是最基本的一点！你剪掉的分支，必须在数学上** 100% 绝对不可能诞生答案 **。如果剪枝条件写得太激进，把原本隐藏的正确答案也顺带剪掉了，代码就会直接爆出 Bug。
