6.7 递归算法复杂度分析

⏱️ 魔法时钟与空间指环：

当你念出一段递归咒语时，它会在一瞬间华丽地完成，还是会让整个世界陷入漫长的等待直到宇宙毁灭？它会安静地运行，还是会瞬间撑爆你的魔法指环（内存）导致“栈溢出”？

为了掌控魔法的威力，我们必须学会给递归算法把脉！今天，我们将学习如何看透递归的 时间复杂度（运行速度） 和 空间复杂度（内存消耗）。

一、递推方程与递归树方法

评估递归快慢的第一步，是把代码翻译成数学语言。我们通常用 T(N) 来表示解决一个规模为 N 的问题所花费的时间。

递推方程的万能公式：

T(N) = a * T(N/b) + 本层的工作量

  • a： 你把问题劈成了几个子问题？（比如归并排序一分为二，a就是2）
  • N/b： 每个子问题的规模缩小了多少？（比如对半劈开，b就是2）
  • 本层的工作量： 除了抛给小弟的任务，你自己（老板）还要花多少时间合并或者处理？（比如遍历一遍数组，就是 O(N)）

如果公式看起来太抽象，我们可以把它画成一棵 “递归树”。把每一层老板和小弟们干的活加起来，就是总的耗时！

二、老板与打工人的巅峰对决：主定理

每次画树太累了！前人帮我们总结出了一套口诀，只要套进公式，三秒钟就能看出递归的时间复杂度，这就是大名鼎鼎的 主定理 (Master Theorem)。

我们依然把递归看成 “老板（根节点合并工作）” 和 “打工人（叶子节点碎活）” 之间的效率拔河比赛。看谁干的活更多，谁就决定了整个公司的最终耗时。

  1. 👑 老板太累了 (合并占大头)： 如果大老板在合并数据时花的时间，比所有基层小弟干的活加起来还要多。那整个公司的效率就由老板决定。
    👉 结果：复杂度就是老板合并的时间 O(N^d)。
  2. ⚖️ 势均力敌 (工作量平分)： 如果老板合并花的精力和所有打工人干的活刚好一样多，这就好像每一层都在做同样多的事情。
    👉 结果：每一层代价一样，直接把一层的代价乘以树的层数，即 O(N^d * log N)。（归并排序就是这种情况！）
  3. 🐒 猴子太多了 (叶子占大头)： 如果老板啥事不干（合并极快），但他招了无穷无尽的打工人，最后底下有成千上万个小弟在干碎活。那公司的效率完全被庞大的人数拖垮。
    👉 结果：复杂度就是最底层叶子节点的总数量！

三、常见递归式的秒杀判断

我们用主定理的思维，来秒杀几个最常见的递归算法！

1. 二分查找 (Binary Search)

每次把问题砍掉一半，但我们只去其中一半寻找（另一半直接扔掉）。不需要合并。

  • 递推式： T(N) = 1 * T(N/2) + O(1)
  • 秒杀分析： 老板花 O(1) 的时间看一眼，只有一个小弟在干活，工作量每一层都是 O(1)。符合“势均力敌”的特殊降维版。
  • 最终复杂度： O(log N)
C++ 二分查找 (递归版)
int binarySearch(vector<int>& arr, int l, int r, int x) { if (r >= l) { int mid = l + (r - l) / 2; // 基准情形：老板看一眼就找到了 (代价 O(1)) if (arr[mid] == x) return mid; // 小弟干活：只选择一半去递归！(只有一个分支) if (arr[mid] > x) return binarySearch(arr, l, mid - 1, x); return binarySearch(arr, mid + 1, r, x); } return -1; }
Python 二分查找 (递归版)
def binary_search(arr, l, r, x): if r >= l: mid = l + (r - l) // 2 # 基准情形：老板看一眼就找到了 (代价 O(1)) if arr[mid] == x: return mid # 小弟干活：只选择一半去递归！(只有一个分支) elif arr[mid] > x: return binary_search(arr, l, mid - 1, x) else: return binary_search(arr, mid + 1, r, x) return -1

2. 失忆的斐波那契数列

没有备忘录的纯递归斐波那契 fib(n) = fib(n-1) + fib(n-2)。

  • 分析： 这棵树几乎是不缩水的（n-1 和 n-2 减得太慢），但老板为了把这两个数加起来，只花了 O(1) 的时间。这属于典型的“猴子太多了”结局。
  • 最终复杂度： 指数级爆炸 O(2^N)。这也是为什么算 fib(50) 会卡死的原因。

四、看不见的隐形消耗：空间复杂度与栈深度

评价一个魔法好坏，除了看它跑得多快（时间复杂度），还要看它消耗多少法力值（空间复杂度）。

🧠 递归的内存诅咒：

我们学过“函数调用栈”。递归每往下深入一层，就要在内存里开辟一个“栈帧”小房间。因此，递归算法的空间复杂度，通常等于这棵递归树的最大深度！

避坑指南：
即使是归并排序，虽然它的时间复杂度是 O(N log N)，并且递归调用栈的深度只有 O(log N)，但因为它在“合并”的时候需要额外开辟一个临时的数组来存放排好序的数据，所以归并排序的总体空间复杂度是 O(N)。
这就是为什么在内存极其紧张的嵌入式设备中，大家更偏爱空间复杂度极小（O(log N)）的原地快速排序！
